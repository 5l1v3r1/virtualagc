Introduction

This page is a guide for Virtual AGC Project administrators, describing how to maintain the various online sites associated with the Virtual AGC Project.  However, it is still necessary to have the appropriate administrative credentials to do so, and this page does not give you such credentials.  In general, becoming an administrator requires manual action by existing administrators, and should only be done for trusted contributors having a long-term previous association with the project.

Local-Storage Setup

Because the main Virtual AGC website is stored on a Linux filesystem at ibiblio.org, with Linux filenames and permission structures, and because the helper scripts I provide here are BASH scripts, it would seem prudent only to maintain the main website using a Linux computer or virtual machine as well.  Note that if the advice given on this page is followed completely, approximately 700GB (not a typo!) is currently required, with about 2TB for future expansion.  See the section on Internet Archive backup.  A minimal requirement (no Internet Archive backup) is closer to 50GB plus room for expansion.

In order to maintain the main website conveniently, certain data needs to be maintained in local storage.  To understand this data, one needs to distinguish between development files for the website and downloadable files from the website.  "Development files" are things like HTML and JPG/PNG image files, JavaScript, CSS, and so on, needed to render the web pages correctly in browsers.  These are development files because they tend to change over time, as the website is written and rewritten.  The files themselves are stored on the Virtual AGC Project's GitHub repository, in a special branch named "gh-pages".

"Downloadable files" tend to be files which are very large in size and which never change after they are acquired.  Typically they are digitizations of Apollo-era documents, usually scans which have been converted into PDF format.  Regardless, because they almost never change and because they are very large, they are not stored in the Virtual AGC Project's GitHub repository, which has allocation limits that would constrain storage of this kind of information.  In terms of megabytes, the Virtual AGC website consists almost entirely of these downloadable files, with the development-type files merely being the icing on the cake.

Two local-storage folders are absolutely required for updates to the main website, and two more are "optional" but are required for certain specific types of data:

  • Required development-file directory:  This would be a local clone of the "gh-pages" branch of the Virtual AGC GitHub repository.  By default, it is the directory ~/git/virtualagc-web/.  However, it can be any directory, by storing the name in an environment variable called "HTML".
  • Required website-image directory:  This would be an exact image of the desired structure of the main website, except that the development files could be out of date.  (The scripts provided for updating the website copy the development files into this directory before performing any uploads.)  By default, it is the directory ~/Desktop/sandroid.org/public_html/apollo/.  However, it can be any directory, by storing the name in an environment variable called "DOCS".
  • Optional source-tree directory:  This would be a local clone of the "master" branch of the Virtual AGC GitHub repository.  On my own computer it's ~/git/virtualagc/, but it can be any folder, and there's no need for an environment variable.  This directory is necessary if the local administrator wishes to update colorized AGC/AGS software listings.
  • Optional schematics directory:  This would be a local clone of the "schematics" branch of the Virtual AGC GitHub repository.  On my own computer, it's ~/git/virtualagc-schematics, but it can be any folder, and there's no need for an environment variable.  This directory is necessary if the local administrator wishes to update Apollo guidance-system drilldowns or the engineering-drawing search engine.

GitHub Repository

The Virtual AGC Project GitHub repository is found at this link.  This site is used for storing and tracking software source code and other engineering files (such as electrical schematics) subject to development and change.

Most administrative activities should be pretty straightforward, and can generally be understood by consulting GitHub's own documentation.

Some familiarity with the various branches of the repository is useful.  The most-significant branches are:

  • master:  All available Apollo-era software source code (especially the AGC, AGS, LVDC, and Gemini OBC flight computers), plus all assemblers and emulators for them.  LVDC flight programs are not included at the present time because of potential concerns about "exporting" them under ITAR.
  • gh-pages:  Development files (HTML, JPG/PNG images, and similar materials) corresponding to the main project website.  It does not include large, archival objects (such as digitized PDF of Apollo-era documents) that are intended only for download rather than for website development, because those materials are very large in comparison to the GitHub resources allocated to the project.  GitHub presents this material also in a form accessible to browsers, which I call "the staging area" for the main website.  At any given time, the contents of the staging area may or may not be identical to that of the main website, and links on the staging area to downloadable files (available only on the main website) may or may not work.
  • mechanical:  2D mechanical drawings and 3D models created for original Apollo-era mechanical drawings of things like the AGC, DSKY, and so on.
  • scenarios:  Setup data for NASSP/Orbiter of various Apollo mission scenarios.
  • schematics:  Electrical CAD files and simulation files created from AGC electrical schematics.
  • wiki:  Wiki files for the GitHub wiki associated with the project.

Other branches are all, or mostly, created for temporary purposes and don't necessarily have any permanent significance.

To obtain administrative privileges, it's necessary to have a GitHub account, and to have an existing project administrator assign administrative privileges to that account.

Virtual AGC Project Collection on the Internet Archive

The Virtual AGC Project collection at the Internet Archive is found at this link.  This site is used for archival storage of the highest-quality original scans available to us of Apollo-era documents.  In almost all cases, this means that the scans were performed by the Virtual AGC Project, or at its request, without being available elsewhere online.  For example, all original scans of AGC software assembly listings are stored here.  Such scans are generally too large (in megabytes) for convenient day-to-day use.  Thus, smaller, lower-quality versions are created for day-to-day use, and stored at the main Virtual AGC Project website.  Nevertheless, it's desirable to retain the highest-quality scans as well, for purposes such as checking for possible errors in the lower-quality scans, for occasionally providing imagery to graphic artists (or just other people in general) on request, and so on, and that's what our Internet Archive collection is used for.  If you have simply obtained a PDF of a document from someone and want to use it as-is, it would generally be posted directly to the main website, and there would be no purpose in uploading it to our Internet Archive collection.

A slight exception is that a large number of scans (100,000+) of Apollo engineering drawings are stored only in our Internet Archive collection, without being made available in a lower-resolution form on the main project website.

Administrative access to this Internet Archive collection allows you to:

  • Upload new documents to the collection.  An upload will usually, but not always, be a zipfile or tarball of scanned images representing the individual pages of an Apollo-era document.  Occasionally the file might be a PDF of such a document.
  • Change the metadata for the document (using an "Edit" tool on that document's web-page in the collection).
  • Change the specific uploaded files comprising the document (also using the "Edit" tool).
  • Perform other technical operations, such as removing the document from the Internet Archive's index (using the "Manage" tool).

While there are many variations to the number, format, and naming of files uploaded, I would recommend the following:

  • For any given document, upload it as a single zipfile or uncompressed tarball of JPG or PNG images.  Use the highest-quality images available.  PNG is preferable in terms of quality, but if JPG is much smaller without any detectable difference in quality, that's fine too.
  • The page-ordering in which the Internet Archive displays the individual images is determined by the alphabetical order of the filenames.  However, the alphabetical collation may not be what you expect, so it's best to name the PNG or JPG files according to some predictable pattern, such as:  basename_0001.jpg, basename_0002.jpg, ..., with all the page numbers being padded with zeroes to have the same number of digits.  This always results in page-numbering that's exactly what you expect.
  • The zipfile or tarball name must end literally with "_images.zip" or "_images.tar" (note the underline)!  For example, "MyDocument_images.tar".  If you leave out the "_images", then the Internet Archive will not attempt to display the individual pages online, and you'll have to correct the file's name and re-upload it.

When uploading a document, you are asked to specify various metadata about it.  As mentioned above, this metadata can be edited afterwards, but it's usually more convenient to get it right the first time.  Pay particular attention to the following metadata fields:

  • title:  By default, this will be some trivial variant of the filename, and that's usually inadequate, so make it something decent instead.
  • creator:  That's whoever wrote the original document (such as the author's name, or MIT Instrumentation Laboratory, or something of that nature); it's not your name.
  • collection:  That's literally "virtualagcproject" (without the quotes).
  • description:  Self-explanatory.
  • subject keywords:  These will vary, so you'll have to use your own judgement, but it's often some combination of "nasa", "apollo", "agc", "guidance", "computer", etc.
  • language:  Almost always English.
  • license:  Public Domain.  The legal theory here is that material produced under contract for the U.S. Government is not subject to copyright, so it's always in the public domain.  I.e., The People paid to create it, so The People own it; and if everyone owns it, then nobody does.  This reasoning does not apply to commercial material which the government simply chose to purchase after the fact, so if there's reason to believe that that's the case, and that the material has a valid copyright, do not upload it.  Similarly, do not upload material which you believe may be subject to ITAR restrictions.  In terms of material marked as "classified" or "confidential", such material is automatically periodically downgraded in terms of it's classification level, and hence any Apollo-era documents should have been declassified decades ago, regardless of how they were originally marked; nevertheless, it is probably wiser not to upload such material.  (I've not personally seen any such documents, except for a few instances in which they were specifically marked has having become unclassified.)

You can only be added as an admin by a direct action of the Internet Archive itself; i.e., a Virtual AGC administrator cannot make you a Virtual AGC administrator of their own volition, unassisted.  A Virtual AGC administrator should submit a request at info@archive.org for the addition of a specific email address as an extra new administrator.

Internet Archive Backup

One problem with the Virtual AGC Project's Internet Archive collection is that it provides no natural backup mechanism.  With the GitHub site and the IBIBLIO site described above and below, there are natural backup-like mechanisms:  For the GitHub site, many people have local clones of the repository (I certainly do), so the information is inherently redundant, or at least mostly so.  For the IBIBLIO site, administrators need to have complete local clones of the site in order to maintain it; so, again, there are natural backups that are complete or nearly complete.    For the Internet Archive collection, however, there's no such natural backup mechanism, since after a file is uploaded there's no particular necessity to keep a local copy of it, and hence in the event of catastrophic failure of the Internet Archive, there's no reasonable expectation that any of the information uploaded to it is easily recoverable.

It is therefore my own practice as an administrator to maintain a complete local copy of all Virtual AGC Project files from the Internet Archive collection, whether explicitly uploaded by me or whether internally generated by the Internet Archive.  It's a tad wasteful of space, but it's the safest way to go about it.  (I would note that a couple hundred boxes of aperture cards comprising the LM engineering drawings are scheduled for scanning, and while the uploads for them are on the order of 1GB each, the processing performed by the Internet Archive turns them into nearly 10GB downloads.  One might, therefore, argue that a full download is unwarranted.)

This task is simplified by the fact that the Internet Archive itself provides a Python 3 script called 'internetarchive' or 'ia', which which can download Internet Archive files in a very flexible way.  I store these downloads in a dedicated directory called virtualagcproject/, and the download command is simply:

cd virtualagcproject
python3 /PATH/TO/ia download --search 'collection:virtualagcproject'
This creates a subfolder of virtualagcproject/ for each document I've uploaded to our Virtual AGC Project collection, and stores each file (either downloaded or generated) in those subfolders.

Though time-consuming, the downloading process is a very efficient one, in that it splits the download into blocks, and only downloads blocks which haven't previously been downloaded, or whose downloading has failed.  This is helpful, since some files to be downloaded are 1GB+ in size; failure of a download only requires later download of blocks that have failed, rather than attempting to completely download the entire 1GB+.

On the other hand, some care is necessary in interpreting the results of the download operation.  Each document has a printed message indicating whether the result was a "success" or a "failure", but there are many hundreds of documents in the collection already, so it is easy not to notice if there were some failures.  In other words, it may be necessary to re-run 'ia' twice or even more times, but it's necessary first to notice that a failure had occurred, and that's not always so easy.

Note also that due to internal process changes (presumably, improvements) at the Internet Archive, files which were generated ("derived") by the Internet Archive may from time to time be re-derived even though new uploads have not occurred.  Therefore, even when 'ia' has been completely successful, re-running it at some later time may still result in portions of some documents being re-downloaded.

Main Website at IBIBLIO

The main Virtual AGC Project website is at this link, but it is not administered at that link.  Various means are used to administer the site, as follows:

  • Synchronizing the website contents with the contents of local storage: Assuming that the setup instructions for local storage have been obeyed, I do this via the BASH script UpdateApollo.sh, of which there is more discussion below.
  • Individual files can also be uploaded via 'scp', although proper setup of local storage there is little reason to do so, since bulky synchronization is easier and generally requires little more time.  The primary use case, in my experience, would be if there are new documents to upload but a full bulk synchronization isn't immediately desired because the HTML files happen to be in a buggy state.
  • Certain precision changes to the website can be performed directly via 'ssh'.  For example, file permissions can be changed, files can be deleted, etc.
  • Some answers to administrative questions supposedly can be found on the IBIBLIO Interactive Q&A page.  I say "supposedly", because there are generally no answers to questions I have asked there.
  • Help tickets can be filed at the IBIBLIO Help Request page.  I wouldn't count on getting any help.
While you would not normally administer the site using 'ssh', it's helpful to explore how to do that, so that you understand the issues involved in using 'rsync' or 'scp'.  When accessing the site via 'ssh', you find yourself at a Linux command line (at this writing, CentOS 6.10).  Thus any tasks you need to perform at the command line are via normal Linux utilities.  Your "home" directory after login is not where the project website's files are stored, but is basically an empty directory.  The website files, rather, are stored in the directory /public/html/apollo/.

The directory hierarchy of the website is rather complex and inconsistent, having evolved over time without much planning, but in general you can say the following:
  • Most HTML files and PNG/JPG images displayed from HTML pages are stored at the top level (/public/html/apollo/).
  • Most downloadable documents are PDF files stored in the "Documents" directory (/public/html/apollo/Documents/).
  • Colorized HTML assembly listings of AGC and AGS programs are stored in subdirectories of the "listings" directory (/public/html/apollo/listings/).
  • The files for the online search engine for guidance-system engineering drawings are stored in the "TipueSearch" directory (/public/html/apollo/TipueSearch/).
There are many more, but these are the most-immediately relevant in terms of day-to-day maintenance of the website.

For example, you could log in to the directory containing the website files and view the HTML files in that directory with the command-line commands:
ssh username@login.ibiblio.org # Will prompt you for password.
cd /public/html/apollo
ls -l *.html

At this point you could also use 'chmod' to change file permissions, 'passwd' to change your password, and so on.

The same login credentials for 'ssh' are used with 'scp' and 'rsync', and file you upload needs to be sent to /public/html/apollo/ or to subdirectories thereof.  For example, to upload a single file called MyNewDocument.pdf from your current local directory to the Documents directory of the website:

scp MyNewDocument.pdf username@login.ibiblio.org:/public/html/apollo/Documents

But in performing an operation like this, you may find that the uploaded document is not actually viewable from the website.  That could happen, for example, if the file permissions are that it is readable by the owner or the owner and the group, but not by everybody.  Since the browser won't be the file owner or belong to the owner's group, the browser cannot open the file.  This is an example in which cleanup of by an 'ssh' login may be needed afterward.

Problems like that don't occur when you use bulk synchronization of the website to the image of the website on your local storage device(s).  Such bulk synchronization is not normally done using 'rsync' directly, but ranter by using the BASH script UpdateApollo.sh, which automatically takes care of all such permission problems.  Assuming that local storage is set up properly, and that UpdateApollo.sh is in the PATH, the process is very simple; from any directory, simply do:

UpdateApollo.sh

and give the appropriate password when prompted to do so.  From the setup instructions, you'll recall that there are two local directories needed:  $HTML, which is the complete is the local "gh-pages" branch of the Virtual AGC Projects' git source tree, containing stuff like the HTML and PNG/JPG files for the website; and $DOCS, which contains a complete local image of the website, but potentially with the HTML/JPG/PNG out-of-date.  What the script does is to bring the $DOCS directory up to date (by copying in the contents of $HTML), perform various cleanup (such a fixing file permissions, adding .htaccess files), and then uploading the entire contents of $DOCS to the website via 'rsync'.

Useful variants are:
UpdateApollo.sh --dry-run    # Don't actually upload.
UpdateApollo.sh --delete     # Delete remote files that are absent from local storage.

The latter variant (--delete) highlights a danger if there are multiple administrators whose local storage areas are not mutually synchronized.  In that case, administrator A may have files that administrator B does not have; if --delete is used, then successive update operations by administrators A and B will keep removing and adding back the same file.  Thus, multiple administrators must take care to insure that their actions are coordinated and their local storage areas are synchronized.

Here are some other traps into which it is all to easy to fall:

  • The "listings" directory:  Although colorized listings of AGC and AGS code are created by building the Virtual AGC Project software from source, those colorized HTML files simply remain in the build directories.  In other words, they don't automatically get copied into the local image of the "listings" directory, and hence they don't automatically get updated on the website.  It's necessary to explicitly copy the colorized HTML files from the project's build directories into the website's "listings" directory.  I'll illustrate from my own computer's setup.  There, the "master" branch of the Virtual AGC source tree is in the folder git/virtualagc/, while the "gh-pages" branch (i.e., the website's HTML/PNG/JPG) is in the git/virtualagc-web/ folder.  If there's a new AGC software version available that has never appeared online before, coming (say) from the source directory git/virtualagc/Artichoke123/, then you'll need to explicitly create a target directory for it the first time it's going to be used:
mkdir git/virtualagc-web/listings/Artichoke123
cd git/virtualagc/Artichoke123/Apollo32.png
In general, though, to simply update the "listings" directory using existing listings, you just do this:
cd git/virtualagc
find -name "*.agc.html" -exec cp {} ../virtualagc-web/listings \;
Having done this, an UpdateApollo.sh operation now sends up-to-date listings to the website.  Of course, given that you are working in the local clone of the "gh-pages" branch of the git repository, you also need to use whatever git commands are appropriate for adding any new directories to the repository or for committing the newly-changed HTML files to the repository.  Note that while the operation above has overwritten all of the listings/*.html files, the new files are almost always identical to the older ones they replaced, and hence committing them has no negative effect on bandwidth or storage allocated in the git repository.
  • The "TipueSearch" directory:  This directory, as already mentioned, contains files for the online search engine for Apollo guidance-system engineering drawings.  The directory is updated by running the script called "makeAllDrilldowns.sh" from the DrawingTree/ directory of the "schematics" branch of the Virtual AGC git repository.  Running that script also updates all of the HTML files (NNNNNNN-NNN.html) associated with "drilldowns" of Apollo guidance-system assemblies.  Thus, if changes had been made to the DrawingTree/ directory of the "schematics" branch, one might want to run "makeAllDrilldowns.sh" prior to UpdateApollo.sh.

The traps mentioned above are also problematic when there are multiple administrators for the website, since typically you'd expect only one of them, rather than all of them, to work on issues related to AGC assembly listings or to Apollo guidance-system assemblies.  So not all of them would be keeping the associated files up-to-date in their local storage.

I have requested information from IBIBLIO, both by Interactive Q&A and Help Ticket, as to how to add additional administrators for the Virtual AGC Project's collection at IBIBLIO.  Alas, I do not expect to receive any help, though if I do then I'll update the description here.  At present, there is a single administrative account, with username "apollo" and a single associated password.

Mailing List at IBIBLIO

The mailing list is administered at this link.  As an admin, you can administer the obvious stuff, such as mailing-list settings, user lists, and so on.

Administrative access to the main website as described above does not give you administrative access to the associated mailing list.  A mailing-list administrator can add other mailing-list administrators at the link above, and then inform the new administrators of the administrative password.

Warning:  All that happens when an administrator is added is that the new email address begins to receive email messages targeted at administrators; it does not allow the new administrator to actually log in and administer anything.  Rather, the administrative interface has a single password (same password for all administrators!), and anybody with this password can log into the administrative interface and administer anything they like, regardless of whether they've been registered as an administrator or not.  Thus, anybody with the password, administrator or not, could log in, change the password, and lock out all of the registered administrators.  Or a registered administrator could do so, and lock out all of the other registered administrators.  Indeed, I'd think this is very likely to happen with a new administrator, simply by accident, if they thought the password they were changing was their own personal password.  If they subsequently forgot the password, thinking there was some easy recover method (there isn't), everybody is locked out!  Poor, poor system.

In addition to list administrators, there is also the separate role of list moderators.  Moderators have elevated permissions and can deal with various problems, but cannot change some of the configuration settings that administrators can.  Most importantly, there is a separate moderator password from the administrator password, and therefore (where possible) it is preferable to give moderator status rather than administrative status.
